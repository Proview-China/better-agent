# Praxis Reboot Charter

## 为什么要重启

这次重启的前提很简单：旧实现已经不再适合作为继续迭代的地基。

我们不再把精力花在“继续缝补原有结构”上，而是先把边界、职责和验证路径想清楚，再重新搭建项目。

## 这条分支的工作原则

1. 先定义边界，再写实现。
2. 先做小而可验证的骨架，再补复杂能力。
3. 每个模块都要有明确职责，不做“什么都能往里塞”的大杂烩。
4. 提交必须能解释清楚意图和验证方式，不靠感觉推进。

## 第一批必须回答的问题

1. 这个项目的最小可行内核到底是什么。
2. 内核、工具运行时、UI/宿主层之间如何解耦。
3. 状态、事件、消息、执行结果分别由谁拥有。
4. 哪些能力是核心内建，哪些能力必须通过插件或外接层接入。
5. 测试链路如何从最小单元一路通到端到端验证。

## 推荐的第一阶段里程碑

1. 写出新的顶层架构说明和模块边界图。
2. 定义核心接口草案，包括输入、输出、错误和事件模型。
3. 再根据接口反推目录结构与基础工程脚手架。
4. 用一个最小闭环能力做第一条实现主线，验证新骨架不是纸上谈兵。
