## 新增需求

### 需求:记忆能力必须以 core API 形式暴露给 UI
记忆模块必须由 `better-agent/core` 暴露稳定的 C API / JSON 接口供 UI 调用。UI 可以提交原始输入、查询条件与展示限制，但禁止在 UI 层实现记忆归一化、冲突裁决、排序或注入裁剪逻辑。

#### 场景:UI 写入一条记忆输入
- **当** macOS UI 或其他 UI 层向 core 提交一条记忆写入请求
- **那么** core 必须返回完整的写入结果、标准化条目或错误对象
- **那么** UI 不需要自行补全 memory entry 结构

#### 场景:UI 查询注入候选
- **当** UI 向 core 查询记忆注入候选
- **那么** core 必须返回已经排序并裁剪后的候选集
- **那么** UI 只负责展示或转发结果，而不是再次执行排序与冲突判断

### 需求:接口错误模型必须与现有 core 风格一致
memory 相关接口必须沿用现有 core 的风格：C 兼容签名、JSON 输入/输出、错误对象可诊断、边界处不直接抛异常。失败结果必须给出明确错误码，而不是只返回模糊文本。

#### 场景:memory 查询请求格式错误
- **当** UI 传入不合法的 JSON 或缺失必要字段
- **那么** core 必须返回结构化错误对象
- **那么** 错误对象必须包含错误码与原因说明

### 需求:绑定层必须保持薄封装
Node binding 与其他绑定层在记忆模块上必须只做薄封装，不得重写 core 中已经存在的记忆规则。绑定层可以负责参数传递与字符串编解码，但禁止自行实现 memory guard 或 retrieval 逻辑。

#### 场景:Node binding 暴露 memory 查询
- **当** Node binding 调用 memory 查询接口
- **那么** 它必须直接返回 core 的 JSON 结果
- **那么** binding 层不得重新计算排序、裁剪或冲突状态
