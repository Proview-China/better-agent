## 新增需求

### 需求:Memory Retriever 必须提供可解释的检索结果
core 必须支持按任务意图、主题、作用域与 layer 检索记忆条目，并返回可解释的命中原因。检索结果必须包含排序后的候选条目以及至少一种解释字段，例如命中的 topic、scope、layer 或证据匹配原因。

#### 场景:按任务意图检索
- **当** UI 提交带有任务意图与主题关键词的查询
- **那么** core 必须返回排序后的记忆候选集
- **那么** 每条候选结果必须能说明命中原因

### 需求:Memory Guard 必须处理过期、冲突与低置信条目
core 必须在检索与写入路径中处理过期、冲突与低置信条目。过期条目默认不得进入注入结果；当实时证据与旧记忆冲突时，core 必须以实时证据为准并回写修正或降权旧记忆；低置信条目必须在排序中降权。

#### 场景:过期条目被查询
- **当** 一条记忆已经过期
- **那么** 默认检索结果不得把该条目作为可注入候选返回
- **那么** 如需返回该条目，结果中必须明确标识其已过期

#### 场景:新证据与旧记忆冲突
- **当** 新输入明确与已有记忆冲突
- **那么** core 必须优先采用新证据
- **那么** 旧记忆必须被标记为冲突、降权或 superseded

### 需求:注入结果必须受上限控制
core 必须支持以条目数、总字符数或等价预算控制单次注入规模，避免再次挤爆上下文。排序与裁剪必须在 core 内完成，UI 禁止自行重排后再决定最终注入集合。

#### 场景:查询结果超过注入上限
- **当** 命中的记忆条目数量超过配置的注入上限
- **那么** core 必须按排序规则裁剪结果
- **那么** 返回结果中必须说明发生了裁剪
