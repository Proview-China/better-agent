## 新增需求

### 需求:运行时必须将高价值输入归一化为统一记忆条目
core 必须支持接收工具执行记录与人工/系统结论等高价值输入，并将其归一化为统一的 memory entry。记忆条目必须至少包含 `memory_id`、`topic`、`kind`、`layer`、`scope`、`evidence`、`confidence`、`created_at`、`updated_at` 等核心字段。运行时禁止把大段原始过程噪声不加筛选地直接写入记忆存储。

#### 场景:从工具执行记录写入记忆
- **当** UI 或运行时把一条已完成的工具执行记录提交给 memory ingestor
- **那么** core 必须输出结构化 memory entry
- **那么** 条目中必须保留与该执行记录关联的证据锚点

#### 场景:从人工结论写入记忆
- **当** UI 提交一条人工确认的结论摘要
- **那么** core 必须把该输入归一化为可存储的 memory entry
- **那么** 条目中必须标识来源类型与适用范围

### 需求:Memory Store 必须支持分层存储与本地持久化
core 必须提供 `short_term`、`task`、`strategy` 这类分层存储语义，并且必须把 memory entry 持久化到本地存储，使 core 重启后能够恢复。M0 可以使用本地文件持久化，但禁止只保存在进程内存中。

#### 场景:core 重启后恢复记忆条目
- **当** 一条记忆已经成功写入本地 store
- **那么** core 重启后再次加载 store 时必须能恢复该条记忆
- **那么** 恢复后的字段值必须与持久化内容一致

### 需求:同类记忆必须优先更新而不是无限追加
对于同一主题、同一作用域且可判定为同类的记忆条目，core 必须优先执行更新、合并或 supersede，而不是无限追加重复条目。若发生更新，记忆条目必须保留可追踪的更新时间和被替代关系。

#### 场景:同主题事实被新证据刷新
- **当** 新输入与已有记忆属于同一主题和作用域，且新证据更新了旧事实
- **那么** core 必须优先更新已有记忆或创建 supersede 关系
- **那么** store 中不得无条件保留多条内容等价但版本失控的重复记忆
